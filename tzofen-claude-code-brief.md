# Tzofen · צופן — Build Brief ל-Claude Code

> מסמך זה מרכז את כל החלטות המוצר, המותג והעיצוב עבור Tzofen. תן אותו ל-Claude Code כנקודת אמת (source of truth) לבנייה ולשדרוג.

---

## 0. מה זה Tzofen

אפליקציית **חיזוי משחקי כדורגל לקהילה** (ליגת העל / הליגה הלאומית הישראלית). המשתמש צובר **נקודות וירטואליות** ומתחרה בלייב מול משתמשים אחרים, ומשלם על תוכן ותחזיות AL מתקדמות.

**חשוב — אין הימורים ואין כסף אמיתי בין משתמשים.** אין הפקדות, אין משיכות, אין הימור. כל "win" הוא נקודות וטבלה בלבד. כל השפה, הקופי והעיצוב חייבים להתרחק מעולם ההימורים (אסור: "Bet", "בנקרול", "הימור", "הפקדה").

**מודל הכנסה:** מנוי (חודשי/שנתי) על השתתפות בתחרויות לייב + גישה לתחזיות וניתוחי AI.

## 2. הלוגו — "שעון המשחק"

קונספט: **זכוכית מגדלת שהעדשה שלה היא שעון משחק.**
- העדשה = לוח שעון; במקום הספרות 12/3/6/9 — **תוצאות כדורגל** (2:1, 1:0, 3:2, 0:0).
- המחוגים בתנוחת **10:10** (התנוחה ה"מחייכת" והמזמינה).
- במרכז, כציר השעון — **כדור כדורגל קטן**.
- ידית אלכסונית עבה מעגנת את ההרכב (תחושת זכוכית מגדלת).

**כללי שימוש:**
- הלוגו חייב **למלא את הריבוע** של אייקון האפליקציה מקצה לקצה — לא "לשחות" באמצע.
- גרסה מלאה (עם תוצאות) לגדלים גדולים; גרסה מפושטת (תוצאות → נקודות זוהרות) ל-favicon וגדלים קטנים (≤64px).
- קובץ קונספט קיים: `tzofen-logo-v2.html` (SVG מחושב ב-JS).

## 3. מערכת הצבעים — הפלטה הצעירה

כיוון: **צעיר, נועז, צבעי בסיס בהירים על רקעים מוארים.** לא כהה.

### טוקנים (CSS variables)

```css
:root {
  --blue:   #1F6BFF;  /* ראשי · בית · פעולות ראשיות וקישורים */
  --green:  #13CE66;  /* משחקים · הצלחה · ניחוש נכון */
  --purple: #8B4DFF;  /* AI · פיצ'ר מיוחד */
  --yellow: #FFC91F;  /* גביע · הדגשה · ניצחון (טקסט כהה מעליו תמיד) */
  --red:    #FF3B5C;  /* תחרות/טבלה · וגם שגיאות (הפרדה בהקשר) */
  --white:  #FFFFFF;  /* כרטיסים ומשטחים */
  --ink:    #15151E;  /* כל הטקסט (שחור רך, לא טהור) */
}
```

### Tailwind config

```js
// tailwind.config.js → theme.extend.colors
colors: {
  brand:   { DEFAULT: '#1F6BFF' },
  success: '#13CE66',
  ai:      '#8B4DFF',
  highlight:'#FFC91F',
  danger:  '#FF3B5C',
  ink:     '#15151E',
}
```

### תפקידי הצבעים

- **כחול** — צבע המותג ועוגן הבית. פעולות ראשיות, קישורים.
- **ירוק** — המגרש, "צא לדרך", ניחוש נכון, סטטוס תקין.
- **סגול** — פיצ'ר ה-AI. בפלטה צעירה סגול קורא "חכם/קסום".
- **צהוב** — הגביע, ניצחון, הדגשה. **תמיד עם טקסט כהה מעליו.**
- **אדום** — אנרגיית תחרות (טבלה) + שגיאות. אדום-תחרות = רקע/אקסנט גדול; אדום-שגיאה = ספציפי (כפתור מחיקה, הודעת שגיאה).
- **לבן** — כל הכרטיסים. הניגוד שמכניס אוויר.
- **דיו** — כל הטקסט.

### כלל נגישות — לא לשבירה
> **הצבעים הם לבלוקים, כפתורים ורקעים בלבד. הטקסט תמיד דיו-שחור (`--ink`) או לבן.**
> הרקעים נשארים בהירים מספיק שטקסט כהה ייקרא (יחס ניגודיות ≥ 4.5:1 לטקסט גוף).
> אסור טקסט צבעוני על רקע צבעוני; אסור צהוב כטקסט או על רקע בהיר.

קובץ הדגמה אינטראקטיבי: `tzofen-young-bright.html`.

## 4. מערכת הרקעים המשתנים לפי קטגוריה

**עיקרון מנחה: הרקע זז, החזית קבועה.** הטקסט, הכרטיסים, הלוגו והכפתורים זהים ב-100% בכל הקטגוריות — רק הרקע (3 אורבים מטושטשים בצבע הקטגוריה) והאקסנטים משתנים. זה נותן **התמצאות** (קידוד צבעוני) בלי לשבור אחידות או קריאוּת.

| קטגוריה | צבע דומיננטי | אקסנט | רקע בהיר (tint) |
|---|---|---|---|
| בית | כחול `#1F6BFF` | צהוב | `#EEF3FF` |
| משחקים | ירוק `#13CE66` | כחול | `#EAFBF1` |
| ניבוי AI | סגול `#8B4DFF` | צהוב | `#F3EDFF` |
| גביע | צהוב `#FFC91F` | אדום | `#FFF8E6` |
| טבלה | אדום `#FF3B5C` | סגול | `#FFEEF1` |
| פרופיל | כחול `#1F6BFF` | ירוק | `#EEF3FF` |
| חדשות | אפור-כחול `#3A4A66` | כחול | `#EEF1F6` |

**מימוש:** 3 אלמנטי `.orb` (radial-gradient + `blur(40px)`) מעל קנבס בצבע ה-tint. במעבר קטגוריה — `transition` של `opacity`, `transform` ו-`background` (~0.7–0.9s). אנימציית drift איטית מתמשכת. **חובה לכבד `prefers-reduced-motion`.** רצוי מתג עוצמה ("פופ מלא / עדין").

## 5. דרישות מוצר — 4 השלבים האסטרטגיים

### שלב 1 — מיצוב (בא לידי ביטוי בכל המוצר)
המסר המוביל בכל נכס: **"צוות AI שמפענח לך כל משחק — ואתה מתחרה מול כל ישראל."** ה-AI הוא הגיבור, לא פיצ'ר משני.

### שלב 2 — דיוק מוכח (מטבע האמון)
- **מספר גיבור אחד** בעמוד הבית — דיוק ה-AI לעונה — **תמיד מול קו בסיס** ("מנצח את המועדפות ב-X%").
- מספר **חי ומתעדכן** (לא קבוע): "נותחו 187 משחקים · 61% דיוק".
- **track record שקוף כולל פספוסים** — להציג גם טעויות. בונה אמון.
- **דיוק לפי רמת ביטחון**: "כשה-AI בביטחון גבוה — צודק ב-X%". (משתמש ב-confidence scores הקיימים.)
- **AI מול בני אדם**: "ה-AI מנצח X% מהשחקנים" — מחבר דיוק לתחרות.
- **אזהרה:** עד שיש מספיק נתונים אמיתיים — אסור להמציא מספרים. להציג "בתקופת הרצה — עקבו בזמן אמת".

### שלב 3 — משפך ולולאת חזרה
1. **טעימה לפני חומה** — משתמש חדש רואה פיצוח AI אמיתי למשחק קרוב **בחינם, בלי הרשמה**, ב-30 שניות הראשונות. ה-login הוא חלון ראווה, לא קיר.
2. **ניחוש ראשון מהיר** — יעד ה-onboarding היחיד: להביא לניחוש ראשון מהר (רגע ה"אהה").
3. **Soft paywall** — לבקש תשלום רק כשהמשתמש רוצה יותר (תחרות לייב / ניתוח AI מלא), לא בכניסה.
4. **לולאת חזרה (push):**
   - יום משחק: "עוד שעתיים נסגרים הניחושים של מחזור 12 ⚽"
   - אחרי משחק: "ניחשת נכון! +30 נקודות. עלית למקום 4 בטבלת החברים."
   - **רצף (streak)** עם 🔥: "אל תשבור את הרצף — נשארו 3 שעות."
5. **לולאת צמיחה:** ליגות פרטיות מול חברים · שיתוף לוואטסאפ (ערוץ #1 בישראל) על כל ניחוש/ניצחון/עלייה · תוצאות ה-AI כתוכן ויראלי.

### שלב 4 — ביצועים ונגישות (שכבת ביצוע)
**ביצועים:**
- **Performance budget** — טעינה ראשונית < 2.5s בנייד.
- **טעינה מדורגת** — קודם תוכן וניחוש, אחר כך קישוטים.
- אפקטים כבדים (canvas אצטדיון, particles, confetti, fireworks): **אחד "גיבור" למסך**, לא בו-זמנית. canvas רק היכן שצריך; תמונת fallback למכשירים חלשים.
- מתג "מצב חיסכון" + כיבוד `prefers-reduced-motion`.

**נגישות (WCAG + תקנות ישראל):**
- ניגודיות ≥ 4.5:1 לטקסט גוף.
- לא לסמוך על צבע לבד — הסתברויות AI (ירוק/אפור/אדום) תמיד עם **מספר + תווית טקסט**.
- אזורי מגע ≥ 44×44px (כפתורי ניחוש, נאב תחתון).
- תוויות ARIA תקינות, ניווט מקלדת, סדר לוגי תקין ב-RTL, טקסט ניתן להגדלה.

## 6. דשבורד ניהול (Admin / Ops)

כלי פנימי לצוות. קובץ קיים: `tzofen-admin-dashboard.jsx` (React + Recharts + lucide-react). **מצב כהה** (סטנדרט לכלי עבודה), עברית RTL, אותה שפה ויזואלית.

### מבנה
Sidebar (בצד ימין ב-RTL) + 4 תצוגות:

**1. סקירה כללית**
- 6 כרטיסי KPI: משתמשים רשומים, פעילים היום, הכנסה החודש, MRR, המרה לתשלום, דיוק AI.
- גרף שטח: הכנסות + צמיחת משתמשים (6 חודשים).
- פילוח מקורות תנועה (Pie) — וואטסאפ מוביל (עקבי עם המשפך).
- גרף עמודות: פעילות שבועית (ניחושים ליום).
- פיד פעילות חיה (live).

**2. ניהול משתמשים**
- טבלה: שם, מייל, מנוי, נקודות, דיוק, סטטוס, תאריך הצטרפות.
- חיפוש + פעולות לשורה: צפייה / השהיה / עוד.

**3. תשלומים והכנסות**
- KPI כספיים: הכנסה היום, עסקאות החודש, החזרים, מנויים חדשים.
- טבלת עסקאות: מזהה, משתמש, תוכנית, סכום, סטטוס (שולם/ממתין/הוחזר), תאריך, **כפתור החזר**.
- ייצוא CSV.

**4. מערכת ובקרה ("מאחורי הקלעים")**
- פעולות: הרצת ניתוח AI ידני · סנכרון תוצאות · שליחת Push לכל המשתמשים · מצב תחזוקה.
- ניטור מצב מערכת: API, DB, מנוע AI (Gemini), סנכרון תוצאות.

### דרישות אבטחה (חובה — בצד שרת)
- **הרשאות admin קשיחות** + בקרת גישה (RBAC). הדשבורד מאחורי אימות חזק.
- **לוג ביקורת (audit log)** לכל פעולה רגישה — מי, מה, מתי (במיוחד החזר כספי, השהיית משתמש, מצב תחזוקה).
- **אישור כפול** לפעולות הרסניות.
- חשיפת מיילים/פרטי תשלום רק תחת הרשאה מתאימה.

## 7. Stack & ארכיטקטורה

**Frontend:** React 19 · TypeScript · Vite · Tailwind · Framer Motion · Recharts · lucide-react · RTL · PWA.
**Backend:** Express · tRPC · Drizzle ORM · SQLite/Turso · JWT auth · Zod validation · rate limiting (רב-שכבתי) · DB indexes.
**AI:** ארכיטקטורת multi-agent.
> **מימוש בפועל (נכון ליוני 2026):** הקריאות ל-LLM מנותבות ל-**OpenAI `gpt-4.1-nano`** דרך ה-OpenAI SDK. הפונקציה `callGemini` (ב-`server/_core/gemini.ts`) והקליינטים בסוכנים הם מעטפת על OpenAI — השם "Gemini" הוא שריד היסטורי. סוכן ה-news עדיין מושך מידע "חי" דרך `callGemini` (כלומר OpenAI). מפתח: `OPENAI_API_KEY` / `AI_API_KEY`.

**אינטגרציות:** סנכרון תוצאות דרך 365scores API (cron) + חישוב נקודות.

### סוכני ה-AI
statistics · research · prediction · tactical · news · schedule/fatigue · orchestrator · **points-strategy**.

> **שינוי חשוב:** הסוכן שנקרא בעבר "bankroll management" מוסב ל-**"points-strategy"** (אסטרטגיית הקצאת נקודות/ביטחון). אין בנקרול ואין כסף — המונח הישן הוא שריד הימורי שסותר את המיצוב. לעדכן שם, תיאור וכל קופי נלווה.

**דפוסי חוסן בתזמורת (orchestratorAgent):** `Promise.allSettled` כך שכל סוכן נכשל באופן עצמאי בלי להפיל את כל התחזית · שרשרת **fallback סטטיסטי** (אם ה-LLM לא זמין, מתקבלת תחזית סבירה על בסיס נתוני ה-DB) · ציון **QA** 0–100 לכל פלט סוכן.

## 8. סדר בנייה מומלץ

1. **טוקנים גלובליים** — להחיל את הפלטה הצעירה (סעיף 3) + Tailwind config + כלל הנגישות.
2. **החלפת מיתוג** — GetWinIL → Tzofen בכל מקום (לוגו, נאב, login, מטא, קופי). להסיר שפה הימורית + מגן דוד.
3. **מערכת הרקעים** (סעיף 4) — לבסס לפני בניית עמודים.
4. **המשפך** (שלב 3) — onboarding "טעימה לפני חומה" + ניחוש ראשון מהיר + soft paywall.
5. **דיוק מוכח** (שלב 2) — מודול עמוד בית + דף AI.
6. **לולאות חזרה וצמיחה** — push יום משחק, streak, שיתוף וואטסאפ, ליגות פרטיות.
7. **ביצועים ונגישות** (שלב 4) — לאורך כל הדרך, ולא בסוף.
8. **דשבורד Admin** (סעיף 6) — כולל שכבת ההרשאות והלוג בצד שרת.

## 10. עדכוני הנדסה ואבטחה — יוני 2026

> תיעוד מדויק של החידושים שנוספו לקוד. כל פריט מציין את הקובץ, הבעיה שתוקנה וההתנהגות החדשה. כל השינויים נדחפו ל-`origin/main` (ריפו `idantz-cmd/Tzofen`).

### 10.1 חיוב סודות JWT בפרודקשן
**קובץ:** `server/_core/env.ts`
- **לפני:** `JWT_SECRET` ו-`JWT_REFRESH_SECRET` היו `optional` ונפלו בשקט למחרוזות dev קבועות שמופיעות בקוד הציבורי — סיכון לזיוף סשן אדמין אם סוד חסר בפרודקשן.
- **אחרי:** בפרודקשן (`NODE_ENV=production`) שני הסודות **חובה, באורך ≥32 תווים**, והאפליקציה **מסרבת לעלות** (`process.exit(1)`) אם הם חסרים. ה-fallback ל-dev נשמר רק מחוץ לפרודקשן.
- **דרישת deploy:** יש להגדיר בסביבת הפרודקשן גם `JWT_SECRET` וגם `JWT_REFRESH_SECRET` (מחרוזות אקראיות שונות ≥32 תווים) — אחרת השרת לא יעלה.

### 10.2 Rate limiting רב-שכבתי לסוכני ה-AI
**קובץ:** `server/_core/index.ts`
- שלוש שכבות: `globalLimiter` (200 ל-15 דק' על `/api/trpc`) · `authLimiter` (10 ל-15 דק' על login+register) · `aiLimiter` (**10 לדקה** על פרוצדורות `agents.*`).
- **תוקן באג:** `app.use("/api/trpc/agents", aiLimiter)` מעולם לא נתפס — tRPC מנתב פרוצדורות כנתיב שטוח עם נקודה (`/api/trpc/agents.query`), לא כסגמנט מקונן, כך שה-endpoint היקר ביותר היה ללא הגנת קצב.
- **הפתרון:** middleware על `/api/trpc` שבודק את שמות הפרוצדורות (כולל בקשות batched כמו `leaderboard.top,agents.query`) ומפעיל את `aiLimiter` כש-`agents.*` מעורב. **אומת חי:** בקשה 11 מחזירה `429`; מסלולים שאינם agents אינם מושפעים.

### 10.3 ולידציית Zod לתשובות JSON של ה-LLM
**קבצים:** `server/agents/llmSchemas.ts` (חדש) · `deepPredictionAgent.ts` · `orchestratorAgent.ts`
- **לפני:** `JSON.parse` ואז פיזור שדות ישירות ל-`DeepMatchPrediction`. ערך שגוי-סוג (למשל `"homeWinProb": "high"`) אינו `null/undefined`, אז `?? fallback` לא תפס אותו — והוא השחית את התחזית בשקט.
- **אחרי:** סכמות Zod (`llmDeepPredictionSchema`, `llmSynthesisSchema`) שבהן כל שדה הוא `.optional().catch(undefined)` — ערך שגוי-סוג / מחוץ-לטווח / enum לא חוקי קורס ל-`undefined` כך שה-fallback הסטטיסטי לכל שדה נכנס לפעולה; payload שאינו אובייקט → `{}`. `parseLlmJson()` גם מנקה code-fences של markdown.

### 10.4 תיקון קריסת מיגרציות (Drizzle)
**קובץ:** `repair-migrations.mjs` (חדש)
- **הבעיה:** מסד הנתונים המקומי הכיל את כל הטבלאות אך טבלת המעקב `__drizzle_migrations` הייתה ריקה → Drizzle ניסה להריץ מחדש את `0000` וקרס ב-`table users already exists`, לפני שהשרת בכלל עלה.
- **הפתרון:** סקריפט שמחשב את ה-hash בדיוק כפי ש-Drizzle מחשב, מוודא שהטבלאות באמת קיימות, ומסמן את המיגרציה כמיושמת ב-`__drizzle_migrations`. **idempotent** (בטוח להרצה חוזרת). אחרי הרצתו השרת עולה נקי.
